# Quick Start

One-shot initialization that takes a brownfield project from zero MindContext to ready-to-code in a single pass.

**No questions asked** - analyzes codebase and generates everything automatically.

## When to Use

- User says "quick start", "bootstrap", "full init", "set up everything"
- User wants to skip the conversational workflow
- Brownfield project where codebase can be analyzed
- User provides a feature/goal to work on

## Workflow

### Step 1: Parse Input

Extract the feature/goal from user input:
- "quick start **add user authentication**" â†’ feature = "user-authentication"
- "bootstrap **dark mode**" â†’ feature = "dark-mode"
- If no feature specified, ask: "What feature do you want to work on?"

### Step 2: Analyze Codebase

Run all analysis in parallel (no user interaction):

```bash
# Project info
cat README.md 2>/dev/null | head -100
cat package.json 2>/dev/null
cat pyproject.toml 2>/dev/null
cat Cargo.toml 2>/dev/null
cat go.mod 2>/dev/null

# Structure
find . -type d -not -path '*/\.*' -not -path '*/node_modules/*' -maxdepth 3 2>/dev/null
ls -la src/ lib/ app/ 2>/dev/null

# Git state
git log --oneline -15 2>/dev/null
git branch --show-current 2>/dev/null
git status --short 2>/dev/null
```

### Step 3: Create Directory Structure

```bash
mkdir -p .project/prds
mkdir -p .project/epics
mkdir -p .project/plans
mkdir -p .project/spikes
mkdir -p .project/context
```

## Error Handling

### No Feature Specified
```
"quick start" without feature description:

"What feature do you want to work on?
Example: 'quick start add user authentication'"
```

### Greenfield Project (No Code)
```
"This looks like a greenfield project (no existing code).

Quick-start works best with existing codebases.
For new projects, use 'project init' for guided setup."
```

### Already Initialized
```
"MindContext already initialized (.project/ exists).

Options:
1. Add another feature: 'create prd [feature]'
2. Re-initialize: 'quick start --force [feature]'
3. View current state: 'project status'"
```

## Notes

- Quick-start is opinionated - it makes assumptions
- Generated content should be reviewed before implementation
- Best for getting started fast, then refining
- Works only with brownfield (existing code) projects
- All generated files marked with `quick_start: true` for tracking
